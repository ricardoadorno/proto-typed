# Nomeação e consistência

## Por que importa
Nomes coerentes ancoram telas, componentes e ações. Sem um padrão, fluxos quebram, props ficam ambíguas e a manutenção aumenta. Uma convenção clara facilita a leitura da DSL e garante que o time encontre rapidamente cada elemento.

## Forma canônica (sintaxe mínima)
<DslPreview title="Convenções de nomes para telas e componentes">
```dsl
screen AccountSettings:
  container:
    # Preferências

component AccountCard:
  card:
    ## %title
    > %description
```
</DslPreview>
Por quê: mostra uso de PascalCase para telas e componentes, facilitando referências e reutilização.

## Padrões de uso
<DslPreview title="Props nomeadas de forma descritiva">
```dsl
component BillingSummary:
  card:
    ## %planName
    # %monthlyValue
    >>> Próxima cobrança em %renewalDate
```
</DslPreview>
Quando usar: representar valores específicos do domínio sem abreviações obscuras.
Por quê: nomes explícitos evitam dúvidas ao alimentar dados no componente.

<DslPreview title="Ações com verbos no infinitivo">
```dsl
row-end:
  @ghost[Cancelar](close)
  @[Enviar convite](SendInvite)
```
</DslPreview>
Quando usar: instruir claramente o que acontece ao acionar o botão.
Por quê: verbos consistentes reforçam expectativas do fluxo.

<DslPreview title="Listas com componentes semânticos">
```dsl
component CustomerRow:
  row-between:
    > %customerName
    >>> %lastContact

list $CustomerRow:
  - Ana Souza | Ontem
  - Bruno Lima | Hoje
```
</DslPreview>
Quando usar: nomear componentes de lista com foco no domínio representado.
Por quê: ajuda a mapear rapidamente a estrutura ao navegar pelo arquivo.

## Erros comuns (Do/Don’t)
**Não faça — usar abreviações genéricas**
<DslPreview title="❌ Errado — Abreviação sem contexto">
```dsl
screen Cfg:
  container:
    # Config
```
</DslPreview>

Faça — prefira nomes completos
<DslPreview title="✅ Correto — Nome explícito">
```dsl
screen Settings:
  container:
    # Configurações
```
</DslPreview>
Por quê: nomes completos facilitam a busca e reduzem ambiguidades.

**Não faça — reaproveitar nomes para propósitos distintos**
<DslPreview title="❌ Errado — Mesmo slug para telas diferentes">
```dsl
screen Dashboard:
  container:
    # Métricas

screen Dashboard:
  container:
    # Configurações
```
</DslPreview>

Faça — defina slugs únicos
<DslPreview title="✅ Correto — Slugs distintos">
```dsl
screen Dashboard:
  container:
    # Métricas

screen DashboardSettings:
  container:
    # Configurações
```
</DslPreview>
Por quê: slugs duplicados impedem navegação correta e confundem referências.

**Não faça — misturar idiomas no mesmo fluxo**
<DslPreview title="❌ Errado — Inglês e português juntos">
```dsl
screen Billing:
  container:
    @[Gerar invoice](InvoiceModal)
```
</DslPreview>

Faça — escolha um idioma principal e mantenha-o
<DslPreview title="✅ Correto — Idioma consistente">
```dsl
screen Faturamento:
  container:
    @[Gerar fatura](InvoiceModal)
```
</DslPreview>
Por quê: consistência linguística fortalece compreensão e evita traduções improvisadas.
