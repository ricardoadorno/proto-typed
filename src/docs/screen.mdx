# Screen
## Por que importa
`screen Nome:` é a unidade navegável da DSL. Cada tela organiza conteúdo, conecta fluxos e serve como destino para botões, links ou navegadores. Estruturar telas corretamente garante navegação previsível e manutenção simples.

## Forma canônica (sintaxe mínima)
<DocsCodePreview title="Tela com container principal">
```dsl
screen Home:
  container:
    # Bem-vindo
    > Escolha uma ação para começar.
```
</DocsCodePreview>
Por quê: define uma tela com layout centralizado e hierarquia textual básica.

## Padrões de uso
<DocsCodePreview title="Screen com fluxos complementares">
```dsl
screen Tasks:
  container:
    # Tarefas
    list:
      - Revisar backlog
      - Preparar demo
    row-end:
      @[Criar tarefa](TaskForm)

modal TaskForm:
  card:
    ## Nova tarefa
    ___: Título{Nome da tarefa}
    row-end:
      @ghost[Cancelar](close)
      @[Salvar](saveTask)
```
</DocsCodePreview>
Quando usar: acionar modais a partir da tela principal sem perder contexto.
Por quê: mantém foco na tarefa atual enquanto resolve ações complementares.

<DocsCodePreview title="Screen com múltiplos containers">
```dsl
screen Dashboard:
  container-wide:
    # Indicadores
    grid-3:
      card:
        ## Leads
        # 1.240
      card:
        ## Conversões
        # 134
      card:
        ## Satisfação
        # 92%

  container-narrow:
    card:
      ## Próximas ações
      list:
        - Revisar backlog
        - Agendar retrospectiva
```
</DocsCodePreview>
Quando usar: combinar colunas diferentes na mesma tela mantendo alinhamento consistente.
Por quê: distribui o conteúdo em zonas claras (visualização ampla x ações complementares).

<DocsCodePreview title="Screen com navigator persistente">
```dsl
screen Settings:
  container-narrow:
    # Configurações
    ___email: E-mail{nome@empresa.com}
    row-end:
      @ghost[Cancelar](-1)
      @[Salvar alterações](saveSettings)

  navigator:
    - Home | Home
    - Dashboard | Dashboard
    - Configurações | Settings
```
</DocsCodePreview>
Quando usar: replicar menu inferior em todas as telas do grupo.
Por quê: promove navegação previsível entre seções irmãs.

## Erros comuns (Do/Don’t)
**Não faça — declarar telas com o mesmo nome**
```dsl
screen Dashboard:
  container:
    # Métricas

screen Dashboard:
  container:
    # Configurações
```

Faça — escolha nomes únicos
<DocsCodePreview title="✅ Correto — Nomes distintos">
```dsl
screen Dashboard:
  container:
    # Métricas

screen Settings:
  container:
    # Configurações
```
</DocsCodePreview>
Por quê: slugs duplicados quebram navegação e confundem acionadores.
<DoDont
  dont={{
    title: "Conteúdo solto",
    code: `
screen Home:
  # Bem-vindo
  > Escolha uma opção.
    `,
  }}
  do={{
    title: "Uso de container",
    code: `
screen Home:
  container:
    # Bem-vindo
    > Escolha uma opção.
    `,
  }}
>
  Por quê: layouts (`container`, `stack`, `card`) controlam largura, espaçamento e responsividade.
</DoDont>

## Referências cruzadas

### Relacionados
- [`Navigator`](./navigator): organiza menus que apontam para diferentes screens.
- [`Modal`](./modal): complementa telas com confirmações e formulários rápidos.
- [`Drawer`](./drawer): oferece painéis laterais dentro da mesma tela.
- [`Container`](./container): layout principal aplicado dentro de screens.

### Composições recomendadas
- **Screen + Container** → base para qualquer conteúdo navegável.
- **Screen + Modal/Drawer** → fluxos complementares sem trocar de tela.
- **Screen + Navigator** → navegação persistente entre seções principais.
